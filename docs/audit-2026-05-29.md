## 一、安全性（Agent 系统）

权限模型实际是**三段式**：`agent_service.dart:137` 只有 `effectiveLevel == needsApproval` 才进审批门；`safe` 直接执行、零检查。因此**每个工具的危险等级就是它的安全边界**。

### S1【高】免审批的全盘内容读取与枚举
`search`（unix_search_tool.dart:44 / powershell_search_tool.dart:45）和 `list_directory`（list_directory_tool.dart:21）都是 `safe`，agent_service 直接执行不弹窗；但 `file_read` 却是 `needsApproval`。这自相矛盾：grep 同样泄露文件内容却不用审批。结果是 Agent 可在无任何提示下 `search(type=grep, path=~)` 扫出黑名单外文件里的密钥/令牌，或 `list_directory` 摸清全盘结构。沙箱注释（sandbox.dart:7）声称“准入由用户审批决定”，而这两个工具完全绕过了审批。
**修复方向**：统一读类工具的等级——要么都受审批门控，要么都 `safe` 但在文档/UI 明确这是有意的设计选择。

### S2【高】递归授权可被 `..` 穿越
权限检查在 agent_service 执行前用**原始 args**：`_extractKeyArg` 返回 `args['path']` 原值（permission_service.dart:104），`_matchesPath` 对原值做 `startsWith` 前缀匹配（permission_rule.dart:67-78）；而真正落盘前才 canonical 化（仅在 `sandbox.canWrite` 查黑名单）。对项目目录的一条递归 allow 规则 + `path: /项目/lib/../../../Documents/x` → `startsWith('/项目/lib/')` 为真 → 免弹窗 → 落盘 canonical 到 `~/Documents/x`（不在黑名单）→ 写入成功。即把“已授权目录”提权为“黑名单外全盘”。
**修复方向**：在权限检查**之前**统一 canonical 化路径（解析 `~`、`..`、symlink），使规则匹配与落盘目标一致。

### S3【高】不可信项目级 Skill 提权 + 提示词注入
`skill` 工具是 `safe`（免审批加载，skill_tool.dart:31）；加载后其 `allowed-tools` 会把列出的工具在该上下文降级为 `safe`（skill_registry.dart:93，含 bash/file_write/file_delete）。项目级 skill 从 `Directory.current/.athena/skills` 加载且**覆盖**用户级（skill_registry.dart:26-35），无任何信任确认；其 `description` 还被拼进系统提示词（level1Prompt，skill_registry.dart:38-49）。在不可信仓库里使用 Athena 时，仓库自带的 skill 既能注入系统提示词，又能让危险工具免审批。
**缓解项**：`loadAll()` 仅在 DI 时执行一次，新写入需重启才生效，降低了同会话即时提权，但持久化与跨会话风险仍在。
**修复方向**：项目级 skill 需显式信任确认；或设“危险工具不可被 skill 降到 safe 的下限”。

### S4【中】web_fetch 放开过宽 + 无 SSRF 防护
web_fetch 是 `needsApproval`（好），但其“永久允许”规则走 default 分支生成无 pattern 规则（permission_service.dart:50），`matchesAllow` 对无 pattern 直接返回 true → **一次勾选 = 所有 URL 永久放开**。且 web_fetch 不校验 host（web_fetch_tool.dart:58 仅查 scheme），localhost / 内网 / `169.254.169.254` 云元数据 / POST 任意外发都可达。配合 S1 即外泄通道。
**修复方向**：web_fetch 规则按 host 粒度持久化；拦截 loopback / link-local / 私网段。

### S5【中】审批文案截断隐藏命令尾部
`_formatToolArgs` 把每个参数值截到 120 字符（chat_view_model.dart:927-929）。用户基于截断文本批准，完整命令照跑——`echo <120字符无害>...; rm -rf ~/x` 的尾部不可见。
**修复方向**：shell/命令类审批应展示完整内容（可滚动）。

### S6【中】`forbidden` 等级未被执行
agent_service.dart:137 只判 `needsApproval`；若 `effectiveLevel == forbidden`，审批块被跳过，第 178 行**照常 execute**。当前无 forbidden 工具，属潜伏陷阱：枚举值定义了契约却未强制。
**修复方向**：显式处理 `forbidden → 拒绝执行`。

### S7【中】API Key 明文且可达
`athena.db` 在 Application Support（database.dart:45），存 provider apiKey 明文，**不在沙箱黑名单**（黑名单含 `~/.athena` 但不含该数据目录）。Agent 经 `search`（免审批）或审批后的 file_read 即可读出。
**修复方向**：把应用数据目录纳入黑名单；或密钥转存 OS keychain。

### S8 / S9【低】纵深防御薄弱
shell 黑名单（sandbox.dart canExecute）与 deny 规则（contains 子串）均易绕过：`rm -fr`、`/bin/rm -rf /`（不以 `rm` 开头）、双空格、`tee ~/.ssh/x`（无 `>`）、`>>`、换行第二条命令（`_splitByOperators` 不切 `\n`）、`$(...)`/反引号、`| python`。因 shell 始终需审批，属纵深防御层，但会给“isDangerous 隐藏勾选框”等判断带来假阴性。SQL 全量写日志（database.dart:56）可能含敏感值。

## 二、正确性与 Bug

### C1【高】HTTP client 泄漏
每次调用都 `OpenAIClient.withApiKey(...)` 新建却从不释放（chat_service.dart:19/46/72/97，及 summary/translation/memory/trpg service）。桌面常驻 + Agent 每轮迭代都走 `getCompletion`，未关闭连接累积 → fd/内存增长。
**修复方向**：`try/finally` 关闭，或按 provider 复用单例 client。

### C2【高】取消时内容丢失（已核验）
`_consumeAgentStream` 流式期间只 `_updateMessageInList`（内存，chat_view_model.dart:770/782 等），不落库；落库仅发生在 `_advanceIteration` 的 finalize 与正常结束。取消时 `throwIfCancelled()`（753）在 `assistantMessage = await _consumeAgentStream(...)` 赋值完成前抛出，于是 catch（542）仍用第 512 行的**初始占位**，而非最新 `current`。后果：①本轮已生成文本未落库，刷新即丢；②多轮场景下取消标记被打到第一条消息（可能覆盖已 finalize 的内容），真正在写的占位留空。
**修复方向**：流中增量持久化；catch 基于最新消息落库。

### C3【高·待确认】外键级联可能失效
`foreign_keys` 是连接级开关，代码仅在 init/reset 各设一次（database.dart:63/192）。若 laconic 用连接池/每操作新连接，则 `deleteChat`/`deleteProvider` 依赖的 CASCADE（chat_repository.dart:51 注释明确依赖）在新连接上不触发 → 孤儿 messages/models。`ChatManageService.deleteChat` 有手动补删兜底，但 `ChatRepository.deleteChat` 单独路径没有。
**修复方向**：**先确认 laconic 连接模型**；若非单持久连接，则每连接设 `foreign_keys=ON` 或全部 delete 处显式级联。

### C4【中】裸强转崩溃（已核验）
`getString`→`value as String`、`getInt`→`value as int`（json_map_extension.dart:7,13），不像 getDouble/getBool 做类型兼容。`models.context_window` 经历 INTEGER→TEXT 多次迁移，任一行类型不符即 `TypeError`，整个模型列表加载失败、聊天/设置页瘫。
**修复方向**：改为 `value.toString()` / `int.tryParse` 容错。

### C5【中】翻译流式无输出（已核验）
`performTranslation` 的 `await for` 只 `buffer.write`（translation_view_model.dart:151-156），从不调 `appendTranslatedText` / 写 `translatedText`；`translate()` 还先把它清空（65）。绑定 `translatedText` 的 UI 流式期间全程空白，直到完成后从历史读。
**修复方向**：循环内同步 `translatedText.value = buffer.toString()`。

### C6–C8【中】数据一致性与并发
- **C6**：导入数据（setting_view_model.dart:292）删 provider/model 重插原 ID 但不清/不修 chats → 既有会话 `getModelById` 返回 null 报 “Model not found”。
- **C7**：summary/translation 用 `millisecondsSinceEpoch` 主键（summary_view_model.dart:43）同毫秒碰撞 → 内容写错记录。
- **C8**：TRPG `sendPlayerAction`（trpg_view_model.dart:158）缺 `if(isStreaming.value) return` 重入保护（ChatViewModel.sendMessage:500 有），快速点击并发双流、状态互相覆盖、重复入库。

### C9–C13【低】
- **C9**：retry.dart:43 `$config.maxAttempts` 插值漏 `{}`（日志打成对象，:78 正确可对照）。
- **C10**：`chat_message_service.dart:27-30` 按消息数硬截断可能切断 assistant↔tool 配对致部分端点 400。
- **C11**：`trpg_message_repository.dart:10` 按 `created_at` 排序同毫秒不稳定（messages 表按 id 排序可对照）。
- **C12**：`deleteChat/deleteChats`（chat_view_model.dart:237/263）不检查当前 chat 是否在流式 → 孤儿/无效写入；`renameChat` 后台流（:401-450）不绑定 cancelToken。
- **C13**：`_isRetryable`（retry.dart:91）子串匹配把含 “connection” 的业务错误误判为可重试，最多重试 10 次。

### 正面结论（供复核参考）
迁移均有 `where('name').count()` 幂等守卫；`reset()` 先 DROP TABLE 再重建索引不冲突；`foreign_keys=ON` 刻意置于孤儿清理迁移之后；`updated_at` 维护在各 update 路径已完整覆盖。

## 三、架构与设计

### A1【高】DI 退化为服务定位
di.dart:75-83 全部以无参构造注册，各 VM/Service 用 `param ?? GetIt.instance<X>()`，故生产路径 100% 走 fallback，构造注入只在测试传参时生效。依赖图对编译期不可见。CLAUDE.md “构造注入”与现实不符。
**修复方向**：注册时显式传依赖，让构造参数成为唯一真相源。

### A2【高】VM↔VM 隐藏横向耦合
ChatViewModel 在方法体内 `GetIt.instance<SettingViewModel/ModelViewModel/SentinelViewModel/PermissionService>()`（chat_view_model.dart:177,180,515,711,719,724）；provider/trpg/translation/summary VM 同样。实际依赖 4 个 VM 却不在构造签名，难测、易成运行期环。
**修复方向**：共享逻辑下沉为领域 Service，或显式构造注入。

### A3–A9【中/低】
- **A3**：ProviderVM ⇄ ModelVM 双向驱动刷新（provider_view_model.dart:80,96,115）。
- **A4**：SettingViewModel 直写 ChatService 可变单例 `retryConfig`（setting_view_model.dart:98,208）致配置真相分裂。
- **A5**：“取默认模型”业务逻辑散落 UI 且与 chat_view_model.dart:176-187 重复（sentinel.dart:298-312）。
- **A6**：ChatViewModel 955 行 7+ 职责（含本属表现层的审批弹窗格式化）。
- **A7**：trpg_page.dart:3 死 import Repository（分层泄漏残迹）。
- **A8**：sentinel.dart:307 业务 loading 用 setState 与 VM signal 分裂。
- **A9**：chat 相关 Service 四拆（manage/support/message/service）边界模糊。

**正面**：Repository 层无业务逻辑；Service 层无反向依赖 VM；全项目无 `effect()`，无未 dispose 泄漏。

## 四、文档偏差（CLAUDE.md，D1）

CLAUDE.md 已显著滞后，会误导安全判断与维护：
