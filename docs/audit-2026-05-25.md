# Athena 项目审计报告（2026-05-25）

> 基线 commit：`d305577`
> 审计范围：lib/ 全量代码（166 个 Dart 文件，~22913 行）
> 审计方式：5 路并行 agent（架构、Agent 安全、ChatViewModel 重构、数据库迁移、CLAUDE.md 合规）

## 总览

- **CLAUDE.md 合规性**：无明确违反项（print/setState/平台判断/路由同步均合规）
- **实质性问题**：9 高危 / 8 中危 / 4 低危
- **三大致命问题**：
  1. Agent 安全模型几乎完全失效（PathSandbox 死代码 + Skill allowed-tools 死字段）

  3. 数据库迁移与 reset 无事务保护

## 🔴 高危问题

### A1. PathSandbox 是死代码，所有文件/Shell 工具均无沙箱保护

- **位置**：`lib/agent/permission/sandbox.dart` 全文件无调用方
- **现象**：CLAUDE.md 宣称"PathSandbox 提供路径级隔离"、"文件操作前校验"，但 `grep` 显示 `PathSandbox`/`canRead`/`canWrite`/`canExecute` 仅在 `sandbox.dart` 自身出现。
  - `lib/agent/tool/file_read_tool.dart:43-48` 直接 `File(path).readAsLines()`
  - `lib/agent/tool/file_write_tool.dart:38-40` 直接 `File(path).writeAsString`
  - `lib/agent/tool/file_delete_tool.dart:32-37` 直接 `File(path).delete()`
  - `lib/agent/tool/bash_shell_tool.dart:45-49` 直接 `Process.run('/bin/sh', ['-c', command])`
- **影响**：提示注入即可读取 `~/.ssh/id_rsa`、`~/.aws/credentials`、`/etc/passwd` 等，无任何提示/校验
- **设计注解**：原 PathSandbox 假设"项目根白名单"，是编码 Agent 思路。Athena 定位为**通用 Agent**，工作区是整台电脑而非某个项目目录。修复方案改用「L0 硬黑名单 + L1 全盘审批」两层模型，详见 [fix-plan Phase 1 设计前提](./audit-fix-plan-2026-05-25.md#设计前提athena-是通用-agent不是编码-agent)。
- **状态**：✅ 已修复（Phase 1 批次）

### A2. file_read 是 `DangerLevel.safe` + 无沙箱 = 任意文件读

- **位置**：`lib/agent/tool/file_read_tool.dart:35`
- **现象**：`DangerLevel get dangerLevel => DangerLevel.safe`，`agent_service.dart:120-121` 只对 `needsApproval` 走审批分支
- **影响**：LLM 输出 `{"tool":"file_read","args":{"path":"/Users/cals/.ssh/id_rsa"}}` 即无提示静默执行
- **状态**：✅ 已修复（Phase 1 批次，升级为 needsApproval + L0 沙箱兜底）

### A3. Shell 工具用 `/bin/sh -c` 拼接，黑名单可任意绕过

- **位置**：`lib/agent/tool/bash_shell_tool.dart:45-49`、`lib/agent/tool/powershell_shell_tool.dart:45-49`
- **现象**：
  - 命令直接交 shell 解释，未用 list-form
  - `PathSandbox.canExecute` 黑名单仅 4 项（`rm -rf /`、`sudo `、`chmod 777 /`、`mkfs.`），可被双空格、变量替换、反引号、`$()`、`;`、`|` 等任意绕过
  - `_extractCommandPrefix` 只取首词作为持久化允许规则——`bash` / `cat` 类首词 "Always allow" 后等于全开
  - `workdir` 参数无任何校验，可指向 `/Users/cals/.ssh`
- **影响**：命令注入 + workdir 跳出
- **状态**：✅ 已修复（Phase 1 批次，sandbox.canExecute token-aware + workdir 默认 home + 完整命令字面量授权）

### A4. Skill `allowed-tools` 字段解析但永不消费（虚假权限模型）

- **位置**：`lib/agent/skill/skill_loader.dart:74`
- **现象**：`grep allowedTools lib` 只在 `skill_loader.dart` 中出现 3 次（仅定义与解析），无消费方
- **附带不一致**：CLAUDE.md 写 `~/.athena/permissions.yaml`，代码 `lib/agent/permission/permission_rule.dart:73` 实际读 `.json`
- **影响**：Skill 声明 `allowed-tools: Read, Grep` 仍可调用任意工具（含 shell），安全契约不存在
- **状态**：未修复

### A5. DI 容器形同虚设：注册 LazySingleton 但所有调用方都直接 new

- **位置**：`lib/di.dart:42-51` 注册了 5 个 Repository 为 LazySingleton
- **现象**：所有 ViewModel/Service 均绕过 GetIt：
  - `lib/view_model/chat_view_model.dart:44-48` 直接 `new ChatManageService()` / `ChatSupportService()` / `ChatMessageService()` / `ChatSelectionDelegate()`
  - `lib/view_model/model_view_model.dart:21-22`、`provider_view_model.dart:9`、`setting_view_model.dart:62`、`summary_view_model.dart:15-16`、`trpg_view_model.dart:21-27`、`translation_view_model.dart:14-16`、`memory_view_model.dart:12-16`、`sentinel_view_model.dart:13` 全是直接 `new XxxRepository()`
  - `lib/service/chat_support_service.dart:28-31` 等构造器默认 `?? XxxRepository()` 也是直接 new
- **影响**：LazySingleton 注册的实例从未被取出，单例语义失效；测试无法注入 mock
- **状态**：✅ 已修复（Phase 2 Batch A，补齐 Service/Repository 注册，所有 VM/Service 默认值改为 `GetIt.instance<X>()`）

### A6. Service 反向依赖 Agent（违反 CLAUDE.md 分层方向）

- **位置**：`lib/service/message_send_service.dart:4-5`
- **现象**：CLAUDE.md 分层为 `ViewModel → Agent → Service`，但 `MessageSendService`（Service 层）持有并调用 `AgentService`（Agent 层）。`AgentService` 自身正确依赖 `ChatService`（agent_service.dart:10）。
- **影响**：依赖方向倒置，且 `MessageSendService` 仅做事件 1:1 重打包，没有任何业务增强
- **状态**：✅ 已修复（Phase 2 Batch B，删除 MessageSendService，VM 直接消费 AgentEvent）

### A7. ChatViewModel 重构未完成，核心业务 210 行仍在 VM

- **位置**：`lib/view_model/chat_view_model.dart:497-706` 的 `sendMessage`
- **现象**：占位 assistant 消息落库、多轮迭代切分、buffers/jsonEncode、`_manage.storeMessage`/`updateMessage`/`updateChatTimestamp`、自动重命名触发、错误回滚成 error MessageEntity，**全部 210 行**留在 VM
- **影响**：拆分目标（VM 变薄）未达成；`MessageSendService` 沦为薄包装；`sendMessage` 仍是测试重灾区
- **状态**：✅ 已修复（Phase 2 Batch B，sendMessage 瘦身到 63 行 + helpers 拆分到 _prepareSendContext / _consumeAgentStream / _advanceIteration / _askPermission）

### A8. 流式取消机制脆弱

- **位置**：`lib/view_model/chat_view_model.dart:602`、`lib/page/desktop/home/home_page.dart:194`、`lib/page/mobile/chat/chat.dart:358`
- **现象**：取消仅靠 UI 设 `isStreaming.value = false` 然后 `break`。但：
  - 底层 HTTP/工具调用不会被打断
  - 若卡在 `await onPermission(...)` 弹窗，break 完全不生效
  - `MessageSendService` 拆分后也没暴露 `CancelToken`/`StreamSubscription`
- **影响**：用户点"停止生成"可能只停 UI，后台仍在写库/扣 token/弹权限框
- **状态**：✅ 已修复（Phase 2 Batch B，CancelToken 接入 AgentService 与权限弹窗，stopGenerating 真正打断流）

### A9. 数据库迁移与 reset 无事务保护

- **位置**：`lib/database/database.dart:69-75` `_migrate()`、`lib/database/database.dart:156-172` `reset()`
- **现象**：每条迁移由多个独立 `statement` 组成（如 `202501200001` 含 CREATE/INSERT/DROP/ALTER/REBUILD INDEX 5 步），任一步骤失败留下半成品：
  - `providers_new` 已建但旧 `providers` 仍在
  - 或旧 `providers` 已删而新表未重命名
  - 重启时 `where('name', name).count()` 返回 0 会重新执行 CREATE，遇到 `providers_new` 已存在直接报错
- **影响**：断电/Crash 后数据库可能进入不可启动状态
- **状态**：✅ 已修复（Phase 3 批次，每个迁移与预设逻辑内部包 laconic.transaction，单点失败可独立回滚）

## 🟠 中危问题

### B1. 预设逻辑会重复插入：`providerCount > 0 则跳过`

- **位置**：`lib/database/database.dart:83-153` `_presetProviders` / `_presetSentinel`
- **现象**：用户删除所有 provider/sentinel（合法 UI 操作，`deleteAllProviders` 在 importProviders 流程中使用）后下次启动会全量重插
- **附带**：`migration_202501170001_init` 创建 models 表时未声明 FOREIGN KEY，删 provider 后 models 变孤儿，与新自增 ID 的 provider 错位
- **状态**：✅ 已修复（Phase 3 批次，改用 migrations 表的 `preset_providers_v1` / `preset_sentinels_v1` 标记 + 新迁移给 models 加 FK ON DELETE CASCADE）

### B2. `chat_repository.dart:67-77` 用 SQL 字符串拼接

- **位置**：`lib/repository/chat_repository.dart:67-77` `getChatsAfterId`
- **现象**：
  ```dart
  var sql = '''
    SELECT * FROM chats
    WHERE id > $chatId
    ORDER BY id ASC
    LIMIT $limit
  ''';
  ```
- **影响**：仓库内唯一非占位符 SQL，破坏统一约定，易被复制扩散
- **状态**：✅ 已修复（Phase 3 批次，改用 laconic builder `.where('id', chatId, comparator: '>')`）

### B3. `PRAGMA foreign_keys = ON` 未启用，级联删除注释失效

- **位置**：`lib/database/database.dart`（初始化）、`lib/repository/chat_repository.dart:51-55`（注释"messages 通过外键级联删除"）
- **现象**：`migration_202501170001_init.dart` 声明了 `FOREIGN KEY chat_id REFERENCES chats(id) ON DELETE CASCADE`，但 SQLite 默认每连接关闭外键，未见 `PRAGMA foreign_keys = ON`
- **影响**：`deleteChat()` 后 messages 残留孤儿行
- **状态**：✅ 已修复（Phase 3 批次，启动时 `PRAGMA foreign_keys = ON`，并通过 db_integrity 迁移清理历史孤儿数据）

### B4. `context_window` 类型在迁移间反复变换，导致历史数据语义不统一

- **位置**：`migration_202501200001_fix_providers_models_schema.dart`（TEXT→INTEGER）→ `migration_202511280001_fix_models_schema_types.dart`（INTEGER→TEXT, CAST）
- **现象**：两次迁移之间录入的 `128000`（数字）与预设的 `"128,000 context"`（带千分位与单位的字符串）混在一列
- **影响**：UI 显示与排序均不可靠
- **状态**：✅ 已修复（Phase 3 批次，db_integrity 迁移把纯数字字符串归一为 `"{千分位} context"` 格式）

### B5. `updateModel/Sentinel/Context/Temperature` 不再刷新 `updated_at`

- **位置**：`lib/service/chat_support_service.dart:80-103`
- **现象**：重构前 `_updateChatTimestamp` 私有方法被删除，新 Service 仅 `copyWith` 修改字段，没有更新 `updatedAt`。`ChatManageService.updateChatTimestamp` 只在 `sendMessage` 末尾调用
- **影响**：用户修改模型/温度/上下文后，chat 列表按 `updatedAt` 排序不会冒泡（隐藏的行为变化）
- **状态**：✅ 已修复（Phase 2 Batch B，ChatSupportService 内部 _touchChat 统一刷新 updated_at）

### B6. 错误回滚抹掉 `toolCalls` / `toolResults` / `reasoningContent`

- **位置**：`lib/view_model/chat_view_model.dart:691-702`
- **现象**：catch 分支用 `MessageEntity(id: assistantId, chatId, role:'assistant', content:'Error: ...')` 整体覆盖。若错误发生在已有多轮 tool_call 之后，所有 `toolCalls`/`toolResults`/`reasoningContent` 被清空（构造器默认值）
- **影响**：用户失去工具调用现场，无法重试与调试
- **状态**：✅ 已修复（Phase 3 批次，catch 分支改为 `assistantMessage.copyWith(content: ...)`，保留 toolCalls/toolResults/reasoningContent；错误信息追加到内容末尾）

### B7. UI 组件混入 Agent 层 + Service 层混入 UI 类型

- **位置一**：`lib/agent/permission/permission_dialog.dart` 导入 `flutter/material.dart`、`showDialog`、`router.navigatorKey`
- **位置二**：`lib/service/chat_support_service.dart:13,52-62` `exportImage(GlobalKey)` 操作 `RenderRepaintBoundary`
- **影响**：分层职责混乱，permission_dialog 应在 widget/page 层，exportImage 应回到 VM 或 widget 层
- **状态**：✅ 已修复（Phase 2 Batch A，permission_dialog 移到 lib/widget/；exportImage 渲染逻辑下沉到 widget，Service 仅接 Uint8List）

### B8. `ChatManageService` 与 `ChatSupportService` 职责重叠

- **位置**：`lib/service/chat_manage_service.dart` vs `lib/service/chat_support_service.dart`
- **现象**：
  - 两者都持有 `_chatRepository`、`_messageRepository`、`_providerRepository`
  - 都对 ChatEntity 做单字段更新（manage 的 `togglePin`/`updateChatTimestamp`、support 的 `updateModel/Sentinel/Context/Temperature/renameChatManually`）
  - manage 中 `getModel/getSentinel/storeMessage/updateMessage/refreshMessages` 单行透传 Repository（"假 Service"）
- **影响**：边界模糊，新增方法需纠结放哪
- **状态**：✅ 已修复（Phase 2 Batch B，删除 5 个假 Service 透传方法 getModel/getSentinel/storeMessage/updateMessage/refreshMessages，VM 直走 Repository）
